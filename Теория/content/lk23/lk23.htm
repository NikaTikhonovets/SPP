Тема №23 "Обзор Java EE"

Java EE – широко используемая платформа, содержащая набор взаимосвязанных технологий, которые существенно сокращают стоимость и сложность разработки, развертывания многоуровневых серверных приложений, а также управления ими. Освобождает от системных задач и помогает сосредоточится на представлении данных и логике приложения.

Сервлет – интерфейс Java, реализация которого расширяет функциональные возможности сервера. Сервлет взаимодействует с клиентами посредством принципа запрос/­ответ.

JSP – платформонезависимая, переносимая и легко расширяемая технология для разработки веб-­приложений. Позволяет создавать содержимое, которое имеет как статические, так и динамические компоненты. Код JSP-страниц транслируется в java-код сервлета с помощью компилятора jasper, потом компилируется в байт-код JVM.

EJB может рассматриваться как фреймворк и как компонент. С точки зрения фреймворка EJB – технология, предоставляющая для серверной части WEB-приложения множество готовых решений. С точки зрения компонента EJB – надстройка над POJO-классом, описываемого с помощью аннотаций.

JPA – спецификация Java EE и Java SE, описывающая управление сохранением java-объектов в таблицы реляционной БД в удобном виде.

JSTL – расширение спецификации JSP.

JSF – фреймворк, который облегчает разработку пользовательских интерфейсов для Java EE-­приложений. Он основывается на компонентах, которые сохраняют свое состояние, когда пользователь запрашивает новую страницу и восстанавливается при повторном запросе.

JNDI – что-то наподобие БД, в которой можно хранить информацию или java-объекты. Объекты в нем находятся как файлы в каталогах, т.е, например, каталок+имя файла – идентификатор объекта, по которому можно просматривать данные.

JMS – набор интерфейсов и классов для рассылки сообщений, позволяющих приложениям, выполненных на платформе Java EE, создавать, посылать, получать и читать сообщения.

JTA – набор интерфейсов и классов для транзакций.

JAAS – стандарт API Java EE, который отделяет аутентификацию и авторизацию пользователей от основной программы, чтобы управлять ими независимо от приложения.

Юнит-тестирование - процесс в программировании, позволяющий проверить на корректность отдельные модули исходного кода программы.

Позитивные тесты ставят целью способность системы работать с выполнением тех функций, для которых она разрабатывалась. Как правило, перед началом тестирования создаются тестовые сценарии, которые предусматривают корректное функционирование системы.

Негативное тестирование веб приложений предусматривает отработку тех сценариев, которые могут произойти с системой, если, скажем, пользователь допустит ошибку при вводе данных. В данном случае, система должна быть готова «ответить» на запрос пользователя сообщением об ошибке.
